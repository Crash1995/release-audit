# Release Audit

## Назначение

Используй этот skill только для полного `pre-release` аудита репозитория.
Это не `quick scan` и не `diff review`.

Цель:
- построить инвентарь всего репозитория;
- проверить все человеко-редактируемые файлы;
- явно перечислить `skipped` файлы и причины;
- выдать findings с доказательствами;
- закончить вердиктом `GO` или `NO-GO`.

## Архитектура: координатор + 3 сабагента

Аудит выполняется в три фазы + опциональная фаза cleanup. **Ты — координатор.** Ручное чтение файлов и проверки делегируются сабагентам через Agent tool.

**Обязательно:** перед стартом создай `TaskCreate` по одному на каждую фазу (Phase 0 / Phase 1 / Phase 2 / Phase 5 и, если пользователь просит фиксы, Phase 3 / Phase 4). Помечай `in_progress` при старте фазы и `completed` сразу по завершении — не батчи. Это даёт пользователю видимый прогресс.

### Фаза 0 — Подготовка (координатор)

Выполни последовательно:

1. Прочитай `CLAUDE.md` / `AGENTS.md` в корне репозитория — это project rules.
2. **Проверь наличие прошлых аудитов ДО старта новых сканов:**
   - Посмотри `docs/release-audits/` (или путь, указанный пользователем), найди самый свежий `.md` отчёт по дате в имени.
   - Прочитай его целиком. Выдели из него список открытых findings (P0–P3) и секцию "Осталось" / "До релиза".
   - Посмотри `git diff HEAD` и `git log` с момента прошлого аудита — какие файлы из списка открытых findings изменены.
   - По каждому открытому finding из прошлого отчёта прочитай конкретное место в текущем коде и определи: **закрыт** (проблемы больше нет), **частично закрыт** (изменён, но остался риск) или **не тронут**.
   - Составь короткий список закрытых findings — он пойдёт в секцию "С прошлого аудита → Исправлено" нового отчёта.
   - Если прошлый аудит свежий (< 1 дня) и в коде мало изменений — **всё равно** прогоняй новый аудит (новые findings могут появиться от новых коммитов), но экономь усилия на повторной верификации старых FP, подтверждённых в прошлом отчёте.
3. Запусти инвентарь:
   ```bash
   python "{{SKILL_DIR}}/scripts/inventory_repo.py" "{{ROOT}}"
   ```
4. Раздели файлы на категории: `source`, `config`, `tests`, `docs`, `ci`, `infra`, `generated`, `vendor`, `binary`.
5. Для `generated`, `vendor`, `binary`, `.venv`, `node_modules`, `dist`, `build`, `data`, `notebooks`, `docs`, `.env`, `__pycache__`, кэш-директории, minified — занеси в `skipped` с причиной.
6. Загрузи `.release-audit.toml` если есть:
   ```bash
   python "{{SKILL_DIR}}/scripts/load_audit_config.py" "{{ROOT}}"
   ```
7. **Запусти все pre-scan сканеры разом:**
   ```bash
   python "{{SKILL_DIR}}/scripts/run_prescan.py" "{{ROOT}}"
   ```
   Скрипт возвращает JSON с результатами всех сканеров, сгруппированными по сабагентам:
   - `security` — для security-auditor
   - `source` — для source-auditor
   - `dependency` — для dependency-auditor

8. Подготовь для каждого сабагента:
   - Список файлов для проверки (FILES_LIST)
   - Project rules (PROJECT_RULES — содержимое CLAUDE.md)
   - Pre-scan результаты для его группы (PRE_SCAN_RESULTS)

### Фаза 1 — Параллельные сабагенты

Запусти **все 3 сабагента параллельно** через Agent tool (в одном сообщении, 3 вызова Agent).

Для каждого сабагента:
1. Прочитай промпт-шаблон из `{{SKILL_DIR}}/agents/<name>.md`
2. Замени плейсхолдеры:
   - `{{ROOT}}` → абсолютный путь к ко��ню репозитория
   - `{{SKILL_DIR}}` → абсолютный путь к директории скилла
   - `{{PROJECT_RULES}}` → содержимое CLAUDE.md (или "Не найдены")
   - `{{FILES_LIST}}` → список файлов для этого сабагента
   - `{{PRE_SCAN_RESULTS}}` → JSON-результаты pre-scan для его группы
3. Передай готовый промпт в Agent tool

#### Распределение файлов между сабагентами

| Сабагент | Шаблон | Получает файлы | Pre-scan группа |
|----------|--------|----------------|-----------------|
| **source-auditor** | `agents/source-auditor.md` | Все source файлы (`.py`), кроме tests | `source` |
| **security-auditor** | `agents/security-auditor.md` | Все source + config файлы | `security` |
| **dependency-auditor** | `agents/dependency-auditor.md` | Config + requirements + scripts | `dependency` |

Все 3 агента работают параллельно и независимо. Каждый возвращает JSON с findings.

**ВАЖНО:** Агенты НЕ запускают Python-скрипты — все скрипты уже выполнены в фазе 0 через `run_prescan.py`. Агенты только верифицируют pre-scan находки и выполняют ручную проверку.

### Фаза 2 — Синтез (координатор)

После завершения всех 3 сабагентов:

#### Шаг 1 — Извлечение findings

Каждый сабагент возвращает текстовый ответ с JSON-блоком внутри (обёрнут в ` ```json `).
Извлеки JSON из каждого ответа. Структура:
```json
{
  "agent": "имя-агента",
  "findings": [...],
  "blocked": [...],
  "files_reviewed": [...]
}
```

Если сабагент вернул текст без валидного JSON — занеси в `blocked` с причиной "agent returned unparseable response".

#### Шаг 2 — Confidence split

Каждый finding от агентов содержит поле `confidence: high|medium|low`.
- `high` + `medium` → основной отчёт (группировка по severity)
- `low` → отдельная секция **"Needs Review"** (нумерация `[?-1]`, `[?-2]`)

Low-confidence findings НЕ влияют на release decision и НЕ считаются в severity-счётчиках.

#### Шаг 3 — Дедупликация

Если два агента нашли одну и ту же проблему (совпадение `file` + `line` ± 3 строки + `rule`), оставь finding с более высоким severity и confidence.

#### Шаг 4 — Config overrides

Если есть `.release-audit.toml`, примени `severity_overrides` и `suppressions`.

#### Шаг 5 — Сравнение с прошлым аудитом

Если в фазе 0 был найден прошлый аудит — сопоставь findings: что исправлено, что осталось, что новое.

**Жёсткое требование:** каждый open finding из прошлого отчёта должен попасть ровно в одну из трёх категорий:
- **Исправлено** — в текущем коде проблемы больше нет (подтверждено чтением места). Укажи файл:строку и что именно было сделано (по git diff).
- **Осталось** — проблема всё ещё в коде. Перенеси finding в новый отчёт с той же или обновлённой severity.
- **Понижено / Заменено** — проблема изменилась (например, P1 превратился в P2 после частичного fix, или root cause переехал в другой файл). Объясни связь.

Не оставляй findings из прошлого отчёта без явного статуса. Если новый отчёт не содержит пункта "Осталось" по конкретному прошлому finding — значит в новом отчёте должно быть сказано, что он **закрыт**.

#### Шаг 6 — Release decision

Примени NO-GO Policy (см. ниже).

#### Шаг 7 — Coverage

- `total files` = все файлы из инвентаря
- `reviewed files` = объединение `files_reviewed` от всех сабагентов (дедуплицированное)
- `skipped files` = файлы из skipped-списка (фаза 0)
- `blocked checks` = объединение `blocked` от всех сабагентов

#### Шаг 8 — Отчёт

Сохрани Markdown-отчёт в `docs/release-audits/YYYY-MM-DD-HHMMSS-release-audit.md` по шаблону из `references/report-template.md`.

### Фаза 5 — Cleanup аудит-артефактов (координатор)

**Обязательная финальная фаза.** Сразу после сохранения нового отчёта — почисти папку `docs/release-audits/` (или путь, указанный пользователем) от устаревших артефактов.

#### Правила cleanup

- **Оставь:** новый отчёт, который ты только что сохранил.
- **Оставь:** максимум **один** предыдущий отчёт, **если** в новом отчёте есть нерешённые открытые findings и прошлый отчёт нужен как полный контекст. Если все открытые findings из прошлого отчёта закрыты в новом — предыдущий отчёт удаляется.
- **Удали:** все `.json` артефакты (они избыточны, информация уже в `.md`).
- **Удали:** все остальные `.md` отчёты старше самого свежего прошлого.
- **Удали:** временные файлы (`/tmp/inventory.json`, `/tmp/prescan*.json`, `/tmp/*_files.txt`), которые ты создал в фазах 0-2.

#### Процесс

1. Выведи команду `ls -la` целевой папки — пользователь увидит что там лежит.
2. Сформируй явный список файлов к удалению и **покажи его в тексте перед удалением** (одна строка: "Удаляю: файл1, файл2, ...").
3. Выполни `rm -v` по явному списку — только именованные файлы, **никогда** `rm -rf` или wildcards.
4. Выведи `ls -la` после удаления для проверки.
5. Почисть `/tmp/` артефакты.

#### Когда НЕ удалять

- Если пользователь явно попросил сохранить историю — оставь все.
- Если прошлый отчёт содержит нерешённые P0/P1 и новый аудит это подтверждает — оставь для полного контекста.
- Если в папке есть файлы, которых ты не создавал и формат которых не распознаётся как release-audit (`YYYY-MM-DD-*-release-audit.{md,json}`) — не трогай их.

#### Принцип "актуальное и недоделанное"

Цель cleanup — держать в папке только то, что реально нужно для будущей работы: один свежий отчёт с полной картиной. Если все прошлые findings закрыты и новый отчёт зафиксировал прогресс в секции "С прошлого аудита → Исправлено" — старые отчёты больше не источник истины, их можно удалить. История прогресса остаётся в самом последнем отчёте.

## Project-specific режим

- Если в репозитории есть `AGENTS.md`, прочитай его в начале аудита и зафиксируй локальные правила.
- Если проект Python/Web3-ориентированный, дополнительно ищи нарушения правил по `Decimal`, `loguru`, `tenacity`, `.env`, таймаутам HTTP и логированию секретов.
- Если в корне есть `.release-audit.toml`, загрузи его и примени `severity_overrides` и `suppressions` до финального verdict.
- Если локальные правила не найдены, оставайся на общих release-критериях и явно скажи об этом в допущениях.

## Области проверки

Проверь и зафиксируй findings по этим блокам:
- заглушки, task markers, временный код, захардкоженные ответы, закомментированный мёртвый код;
- безопасность: секреты, небезопасный shell, небезопасная десериализация, `eval`/`exec`, опасные конфиги;
- утечки данных: токены в browser storage, чувствительные поля в логах, `.env`/gitignore риски;
- память и производительность: blocking calls, тяжёлые вызовы в циклах, незакрытые ресурсы, listener/resource leaks;
- полнота реализации, если найден источник требований в `README`, `docs/`, `spec`, `issues`, `PRD`;
- обработка ошибок и `edge cases`;
- качество кода: длина функций, вложенность, дублирование, типизация, naming;
- технический долг: пустые функции, task markers в source-коде, слабые имена, mutable defaults, blocking calls в async-коде;
- зависимости, конфигурация окружения и риски утечки через git;
- готовность к продакшену: debug-флаги, логирование, graceful shutdown, таймауты, лимиты;
- cleanup репозитория: устаревшие docs, legacy-конфиги, backup-файлы, неиспользуемые тесты и служебный мусор.

## Coverage

- В отчёте обязательно покажи `total files`, `reviewed files`, `skipped files`, `blocked checks`.
- Если файл человеко-редактируемый, он должен попасть либо в `reviewed`, либо в `blocked` с причиной.
- Если репозиторий большой, проверяй пакетами по категориям, но не завершай аудит до полного покрытия.

## Формат findings

Для каждого finding укажи:
- `category`;
- `severity`;
- `file:line`;
- краткое название проблемы;
- почему это риск;
- доказательство из контекста;
- минимальное исправление.

Если проблема зависит от окружения или не может быть подтверждена локально, перенеси её в `Blocked` вместо уверенного finding.

## Финальный ответ

Отчёт для пользователя — **человекочитаемый Markdown**, не JSON-дамп. Следуй шаблону из `references/report-template.md`. Порядок секций:

### 1. Verdict (первая строка)
Одна строка: `GO` или `NO-GO` + причина NO-GO + дата + счётчик файлов.

### 2. Summary (3-5 строк)
Что за проект, сколько findings по severity, главный риск или его отсутствие.

### 3. Findings по severity
Группировка: сначала severity (`P0` → `P1` → `P2` → `P3`), внутри — по категории.
Каждый finding — компактная карточка:
- Сквозная нумерация: `[P1-1]`, `[P1-2]`, `[P2-1]`
- Файл:строка
- Цитата кода (1-3 строки)
- Fix (одно предложение)

Категории: `Bugs and Logic Errors`, `Security`, `Performance and Memory`, `Data Leaks`, `Code Quality`, `Technical Debt`, `Dependencies and Configuration`, `Cleanup`.
Если в категории нет находок — одна строка `No issues found.`

### 3.5. Needs Review (low confidence)
Findings с `confidence: low` — отдельная секция, нумерация `[?-1]`, `[?-2]`. Не влияют на verdict.

### 4. С прошлого аудита
Три списка: исправлено / осталось / новое.

### 5. До релиза
Чеклист-формат: обязательно / желательно (можно после релиза).

### 6. Cleanup-кандидаты
Отдельный блок — не блокирует релиз, но стоит убрать.

### 7. Coverage
Таблица: всего / проверено / skipped / blocked / сабагенты.

### 8. Blocked & Допущения
Что не удалось проверить и почему.

---

Если пользователь не просил исправления, не переходи к патчам автоматически. Сначала закончи аудит.

Если пользователь просит исправить найденные проблемы (или сказал заранее "исправь всё") — переходи к Фазе 3.

### Фаза 3 — Fix agents (после отчёта, по запросу пользователя)

После синтеза отчёта, если пользователь подтвердил исправления:

#### Шаг 1 — Классификация findings по группам

Раздели все findings (P0–P3) на 3 группы:

| Группа | Что входит | Примеры |
|--------|-----------|---------|
| **surgical** | Точечные правки, не затрагивающие логику | bare except → конкретные исключения, HTTP таймауты, .gitignore, requirements.txt, config.yaml, context managers |
| **migration** | Массовые замены по паттерну через много файлов | logging → loguru, ручной retry → tenacity, requests → aiohttp |
| **refactor** | Структурные изменения кода | float → Decimal, декомпозиция функций > 50 строк, дедупликация констант |

Если группа пуста (нет findings для неё) — не запускай агента для этой группы.

#### Шаг 2 — Подготовь промпты

Для каждой непустой группы:
1. Прочитай шаблон из `{{SKILL_DIR}}/agents/fixer.md`
2. Замени плейсхолдеры:
   - `{{ROOT}}` → абсолютный путь к корню репозитория
   - `{{FIX_GROUP}}` → имя группы (surgical | migration | refactor)
   - `{{PROJECT_RULES}}` → содержимое CLAUDE.md
   - `{{FINDINGS_TO_FIX}}` → JSON-массив findings для этой группы

Формат каждого finding в JSON:
```json
{
  "id": "P1-1",
  "severity": "P1",
  "file": "src/collectors/base.py",
  "line": 111,
  "title": "HTTP-запросы без таймаута",
  "fix": "Добавить timeout=aiohttp.ClientTimeout(total=30, connect=5)",
  "evidence": "async with self.session.get(url, params=params) as response:"
}
```

#### Шаг 3 — Запусти fix-агентов параллельно

Запусти все непустые группы **параллельно** через Agent tool (в одном сообщении).

**ВАЖНО:** Группы не должны пересекаться по файлам. Если один файл затрагивается findings из разных групп — отнеси все findings по этому файлу в ту группу, которая делает более глубокие изменения (refactor > migration > surgical).

#### Шаг 4 — Обработка результатов

Каждый fix-агент возвращает JSON с `fixed`, `skipped`, `files_modified`, `compile_check`.
Собери общий результат:
- Объединение `fixed` от всех агентов
- Объединение `skipped` от всех агентов
- Если `compile_check: "fail"` у любого агента — зафиксируй в отчёте

### Фаза 4 — Верификация (verifier-агент + координатор)

После завершения всех fix-агентов координатор запускает verifier-агента.

#### Шаг 1 — Подготовь данные для verifier

Собери из результатов fix-агентов:
- `MODIFIED_FILES` — объединение `files_modified` от всех агентов (дедуплицированный список)
- `BASELINE` — объединение `baseline` от всех агентов (JSON с публичными API и тестовыми результатами до фиксов)

#### Шаг 2 — Запусти verifier-агента

1. Прочитай шаблон из `{{SKILL_DIR}}/agents/verifier.md`
2. Замени плейсхолдеры:
   - `{{ROOT}}` → абсолютный путь к корню репозитория
   - `{{PROJECT_RULES}}` → содержимое CLAUDE.md
   - `{{MODIFIED_FILES}}` → JSON-список изменённых файлов
   - `{{BASELINE}}` → JSON baseline от fix-агентов
3. Запусти через Agent tool

Verifier выполняет 5 блоков проверок:
1. **Compile & Import** — синтаксис и импорты всех изменённых файлов
2. **API Compatibility** — публичные функции/классы не сломаны, переименования не оставили битых импортов
3. **Functional Smoke Tests** — inline тесты ключевых функций с фиксированными входными данными
4. **Regression Check** — сравнение результатов вычислений до/после (если есть baseline)
5. **Cross-module Integration** — проверка что зависимые модули работают с изменённым кодом

#### Шаг 3 — Обработка результатов verifier

Verifier возвращает JSON с `summary.verdict`:
- **PASS** — все проверки пройдены → обнови отчёт, добавь секцию "Verification: PASS"
- **PASS_WITH_WARNINGS** — всё ок, но часть тестов пропущена из-за окружения → обнови отчёт, перечисли что пропущено
- **FAIL** — есть критические ошибки → координатор **не обновляет отчёт как "fixed"**, а:
  1. Читает `failures` из ответа verifier
  2. Пытается исправить проблемы самостоятельно (простые TypeError, пропущенный import)
  3. Если не получается — помечает findings как "fix attempted but verification failed" в отчёте
  4. Перезапускает verifier на исправленных файлах

#### Шаг 4 — Финализация отчёта

1. **Обнови отчёт**: добавь в начало строку "Post-audit fixes applied" со списком исправленных findings.
2. **Добавь секцию Verification** с результатами verifier (summary + детали failures если есть).
3. **Пересчитай статистику**: обнови счётчики P0/P1/P2/P3 с учётом исправленных.

## NO-GO Policy

Автоматический `NO-GO`, если выполняется хотя бы одно условие:
- есть finding c `severity = P0`;
- есть `blocked` проверка по критичному пути;
- найден `hardcoded-secret`, `private-key-material` или `tracked-env-risk`;
- отсутствует `.gitignore` или в нём не закрыт критичный риск утечки env/logs.

Cleanup-находки сами по себе не блокируют релиз, если это не security/runtime риск. Их надо вынести в отдельный блок и явно показать, что можно удалить до публикации.

## Что нельзя делать

- Нельзя ограничиваться `grep`-совпадениями без чтения контекста.
- Нельзя печатать найденные секреты целиком; маскируй значения.
- Нельзя заявлять о полноте реализации, если в репозитории не найден источник требований.
- Нельзя писать "всё проверено", если есть `blocked` проверки или непрочитанные человеко-редактируемые файлы.
- Нельзя пропускать Фазу 0 шаг 2 (чтение прошлого аудита) — иначе рискуешь переоткрыть уже решённые findings и не показать пользователю прогресс.
- Нельзя пропускать Фазу 5 (cleanup) — устаревшие отчёты захламляют папку и мешают будущим аудитам.
- Нельзя удалять отчёты через `rm -rf` или wildcards — только явный список по именам.
- Нельзя удалять прошлый отчёт, если в нём остались нерешённые findings, которых нет в новом отчёте (это означало бы потерю контекста).
